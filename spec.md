# Read-It-Later — Specification

> Spec-Driven Development document. Requirement keywords (**MUST**, **MUST NOT**, **SHOULD**, **SHOULD NOT**, **MAY**, **REQUIRED**, **RECOMMENDED**, **OPTIONAL**) are used as defined in [RFC 2119](https://www.rfc-editor.org/rfc/rfc2119).

---

## 1. Purpose

The system **MUST** provide a minimal personal "read-it-later" application that allows a single user to:

1. Submit a website URL through a web form.
2. Persist the URL together with metadata automatically extracted from the target page.
3. List every previously stored URL on a dedicated page.

The system **MUST NOT** implement authentication, multi-user support, or content synchronization in this iteration. Those concerns are explicitly out of scope (see §13).

---

## 2. Glossary

- **Item** — a single saved record: a URL plus its extracted metadata and bookkeeping fields.
- **Frontend** — the Streamlit application that renders the form and the list page.
- **Backend** — the FastAPI HTTP service that owns persistence and metadata extraction.
- **Store** — the TinyDB JSON file that holds every Item.
- **Metadata** — title, description, image URL, site name, author, and canonical URL extracted from the target page.
- **Add page** — the Streamlit page that hosts the URL submission form; the default page of the frontend.
- **Library page** — the Streamlit page that lists every stored Item.

---

## 3. Technology Stack

The implementation **MUST** use the following stack:

- **Frontend**: Streamlit (Python).
- **Backend**: FastAPI served by Uvicorn (Python).
- **HTTP client (backend → target sites)**: `requests`.
- **HTML parsing**: `beautifulsoup4` with the `html.parser` or `lxml` parser.
- **Persistence**: `tinydb` (single JSON file on disk).
- **Validation**: Pydantic v2 models (bundled with FastAPI).

The frontend **MUST NOT** import the backend's persistence or extraction modules directly. All communication between frontend and backend **MUST** go through HTTP.

The project **SHOULD** declare dependencies in a single `pyproject.toml` or `requirements.txt`. A virtualenv **SHOULD** be used for local development.

---

## 4. Architecture

The system **MUST** be split into two independently runnable processes:

1. **`backend/`** — FastAPI application exposing a JSON REST API on a configurable port (default `8000`).
2. **`frontend/`** — Streamlit application that talks to the backend over HTTP (default `http://localhost:8000`).

The backend base URL **MUST** be configurable via the environment variable `BACKEND_URL` read by the frontend. The frontend **SHOULD** fall back to `http://localhost:8000` when the variable is unset.

The frontend **MUST** call the backend from server-side Python (e.g. `requests`/`httpx`) running inside the Streamlit process; the browser never calls the backend directly. The backend therefore **MUST NOT** be required to expose CORS headers, and CORS middleware **SHOULD NOT** be installed unless a browser-side client is introduced in a later iteration.

A suggested project layout (non-normative):

```
sdd_lab_app/
├── spec.md
├── pyproject.toml
├── backend/
│   ├── main.py           # FastAPI app + routes
│   ├── storage.py        # TinyDB wrapper
│   ├── metadata.py       # URL fetch + HTML parsing
│   ├── models.py         # Pydantic models
│   └── data/items.json   # TinyDB file (gitignored)
└── frontend/
    ├── app.py            # Streamlit entry (form page)
    └── pages/
        └── 1_Library.py  # List page
```

## 5. Data Model

Each Item stored in the database **MUST** contain the following fields:

| Field           | Type             | Required | Notes                                                                 |
|-----------------|------------------|----------|-----------------------------------------------------------------------|
| `id`            | string           | MUST     | UUID v4 generated by the backend.                                     |
| `url`           | string           | MUST     | The original URL submitted by the user.                               |
| `canonical_url` | string \| null   | SHOULD   | Extracted per §7. Stored as an absolute URL (relative values resolved per §7). |
| `title`         | string \| null   | SHOULD   | Extracted per §7.                                                     |
| `description`   | string \| null   | SHOULD   | Extracted per §7.                                                     |
| `image`         | string \| null   | MAY      | Extracted per §7. Stored as an absolute URL (relative values resolved per §7). |
| `site_name`     | string \| null   | MAY      | Extracted per §7.                                                     |
| `author`        | string \| null   | MAY      | Extracted per §7.                                                     |
| `created_at`    | string           | MUST     | ISO 8601 UTC timestamp set by the backend at insert time.             |
| `fetch_status`  | string           | MUST     | One of `"ok"`, `"partial"`, `"failed"` (see §7).                       |

§7 is the single source of truth for metadata extraction precedence and value normalization. If this table and §7 disagree, §7 wins.

Fields marked `SHOULD`/`MAY` that cannot be extracted **MUST** be stored as `null`. The backend **MUST NOT** reject an item solely because optional metadata is missing.

The `url` field **MUST** be validated as a syntactically well-formed `http://` or `https://` URL before persistence. Other schemes **MUST** be rejected with HTTP `422`.

Duplicate URLs **SHOULD** be permitted (the same article may be saved more than once); the frontend **MAY** warn the user but **MUST NOT** block the submission.

## 6. Backend API

The backend **MUST** expose the following endpoints. All payloads **MUST** be `application/json`.

### 6.1 `POST /items`

Create a new Item from a URL.

**Request body**:
```json
{ "url": "https://example.com/article" }
```

**Behavior**:
- The backend **MUST** validate the URL.
- The backend **MUST** attempt to fetch and parse metadata (see §7).
- The backend **MUST** persist the Item even when metadata extraction fails (with `fetch_status = "failed"` and metadata fields set to `null`).
- The backend **MUST** return HTTP `201 Created` with the full persisted Item on success.
- The backend **MUST** return HTTP `422` for invalid URLs with an error body of shape `{ "detail": "<reason>" }` where `<reason>` is a single human-readable string. FastAPI's default validation handler returns a list of objects under `detail`; the backend **MUST** install custom exception handlers for `RequestValidationError` and `ValueError` that flatten the first error's message into the documented shape.
- The backend **MUST** return HTTP `500` for unexpected server errors with a generic message.

### 6.2 `GET /items`

List every stored Item.

- The backend **MUST** return HTTP `200` with a JSON array of Items.
- Items **MUST** be returned ordered by `created_at` descending (newest first).
- The endpoint **MAY** accept `limit` and `offset` query parameters; if implemented, defaults **SHOULD** be `limit=100, offset=0`.

### 6.3 `GET /healthz`

- The backend **SHOULD** expose a liveness probe returning `{ "status": "ok" }` with HTTP `200`.

The backend **MUST NOT** expose any endpoint that mutates or deletes data in this iteration beyond `POST /items`.

## 7. Metadata Extraction

When handling `POST /items` the backend **MUST**:

1. Issue an HTTP `GET` against the submitted URL with:
   - A timeout of `10` seconds (connect + read).
   - A `User-Agent` header identifying the app (e.g. `ReadLater/0.1`).
   - At most `5` redirects followed.
2. Parse the response body with BeautifulSoup **only when** the response status is `2xx` and the `Content-Type` indicates HTML.
3. Extract metadata in the following precedence order for each field:
   - **title**: `meta[property="og:title"]` → `meta[name="twitter:title"]` → `<title>`.
   - **description**: `meta[property="og:description"]` → `meta[name="twitter:description"]` → `meta[name="description"]`.
   - **image**: `meta[property="og:image"]` → `meta[name="twitter:image"]`.
   - **site_name**: `meta[property="og:site_name"]`.
   - **author**: `meta[name="author"]` → `meta[property="article:author"]`.
   - **canonical_url**: `link[rel="canonical"][href]`.
4. Normalize URL-valued metadata:
   - When an extracted `image` or `canonical_url` is a relative URL, the backend **MUST** resolve it against the final response URL (after redirects) using `urllib.parse.urljoin`.
   - After resolution, values that are not absolute `http(s)` URLs **MUST** be discarded (stored as `null`).
5. Strip surrounding whitespace from every extracted string value. Empty strings after stripping **MUST** be stored as `null`.

The `fetch_status` field **MUST** be set as follows:

- `"ok"` — fetch succeeded **and** at least `title` was extracted.
- `"partial"` — fetch succeeded but `title` could not be extracted.
- `"failed"` — the request raised, timed out, returned a non-`2xx` status, or returned non-HTML content.

The backend **MUST NOT** propagate raw exception messages from the target site to the API response. It **SHOULD** log them server-side.

The backend **MAY** cache extraction results in-process **only if** caching is keyed by URL and bounded in size; caching **MUST NOT** persist between restarts.

## 8. Persistence

- The Store **MUST** be a TinyDB instance backed by a single JSON file (default path `backend/data/items.json`).
- The file path **SHOULD** be configurable via environment variable `DB_PATH`.
- The directory containing the database file **MUST** be created on startup if it does not exist.
- Write operations **MUST** be flushed to disk before the API returns `201`.
- The Store **MUST NOT** be wiped or migrated automatically; schema evolution is out of scope for this iteration.
- TinyDB is not thread-safe. The backend **MUST** serialize every write to the Store with a process-wide lock (e.g. `threading.Lock` held for the duration of insert/update). This guards against concurrent `POST /items` calls — including a single user double-clicking Save — corrupting the JSON file.

## 9. Frontend (Streamlit)

The frontend **MUST** provide two pages:

### 9.1 Add page (default)

- **MUST** render a single text input labeled "URL".
- **MUST** render a primary button labeled "Save".
- **MUST** submit the form when the user presses **Enter** while the input is focused. The implementation **SHOULD** use `st.form` with `clear_on_submit=False`; the input is cleared manually after a successful response (see below).
- **MUST** POST the URL to `${BACKEND_URL}/items` with a client-side HTTP timeout of `11` seconds.
- **MUST** reject empty or whitespace-only input client-side: such submissions **MUST NOT** be sent to the backend and **MUST** be reported with an `st.error` "Please enter a URL". No other client-side URL validation **SHOULD** be performed; canonical validation lives in the backend.
- **MUST** display exactly one feedback message after every non-empty submission attempt:
  - `st.success` with the extracted title when the backend returns `201` and `fetch_status` is `"ok"`.
  - `st.info` with the saved URL when the backend returns `201` and `fetch_status` is `"partial"` (item stored but metadata incomplete).
  - `st.warning` with the saved URL when the backend returns `201` and `fetch_status` is `"failed"` (item stored without metadata).
  - `st.error` with the backend's `detail` message when the backend returns `422`.
  - `st.error` with a generic "Backend error, please try again" message when the backend returns any other `4xx` or `5xx`.
  - `st.error` with a generic "Could not reach backend" message when the request itself fails (timeout, connection error).
- **MUST** clear the input only after a successful response (any `201`). On failure the input **MUST** be preserved so the user can retry or edit it.
- **SHOULD NOT** block the UI for longer than 11 seconds (the client-side HTTP timeout).

### 9.2 Library page

- **MUST** be reachable via Streamlit's multipage navigation (e.g. `pages/1_Library.py`).
- **MUST** call `GET /items` on render and display the results.
- **MUST** render each Item as a card showing: title (linking to `url`), description (truncated to ~280 chars), site name, and saved date.
- **MUST** show the raw URL when `title` is missing.
- **SHOULD** display the `image` as a thumbnail when present.
- **SHOULD** show an empty-state message when no Items exist.
- **MUST** display an error message if the backend is unreachable; it **MUST NOT** crash.
- **MAY** offer a "Refresh" button to re-fetch the list.

The frontend **MUST NOT** perform metadata extraction or write directly to the Store.

## 10. User Feedback Contract

Every user-initiated action **MUST** produce exactly one visible feedback message within 11 seconds (10s extraction budget + 1s UI margin). The mapping is:

| Outcome                                       | Severity | Source                                   |
|-----------------------------------------------|----------|------------------------------------------|
| Empty submission                              | error    | frontend client-side check               |
| Saved with full metadata (at least a title)   | success  | backend `201` + `fetch_status="ok"`      |
| Saved with partial metadata (no title)        | info     | backend `201` + `fetch_status="partial"` |
| Saved without metadata (fetch failed)         | warning  | backend `201` + `fetch_status="failed"`  |
| Rejected (invalid URL)                        | error    | backend `422`                            |
| Other client error                            | error    | backend `4xx` (non-422)                  |
| Unexpected backend error                      | error    | backend `5xx`                            |
| Backend unreachable / timeout                 | error    | frontend HTTP client                     |

The frontend **MUST NOT** display silent failures or stack traces to the user.

## 11. Non-Functional Requirements

- **Startup**: each process **MUST** start with a single command (`uvicorn backend.main:app --reload` and `streamlit run frontend/app.py`).
- **Configuration**: every tunable value (ports, DB path, backend URL, frontend origin) **MUST** be reachable via environment variables; hard-coded fallbacks **SHOULD** match the defaults documented above.
- **Logging**: the backend **SHOULD** log every `POST /items` with the URL, resulting `fetch_status`, and duration. It **MUST NOT** log full response bodies of fetched pages.
- **Security**:
  - The backend **MUST NOT** follow redirects to non-`http(s)` schemes.
  - The backend **SHOULD NOT** fetch URLs whose host resolves to a private/loopback range in production; this **MAY** be relaxed for the local development iteration.
  - The frontend **MUST** escape any backend-provided text rendered in the Library to prevent injection through metadata values (Streamlit's default text rendering satisfies this; `st.markdown(..., unsafe_allow_html=True)` **MUST NOT** be used with untrusted strings).
- **Testing**: the backend **SHOULD** ship with at least:
  - A unit test for metadata extraction using a fixed HTML fixture.
  - An integration test for `POST /items` using FastAPI's `TestClient` with the network layer mocked.

---

## 12. Acceptance Criteria

The iteration **MUST** be considered complete when all of the following hold:

1. Starting backend and frontend with the documented commands renders a working Add page on `http://localhost:8501`.
2. Submitting a valid URL to a reachable public page persists an Item with at least a `title` and shows a success message.
3. Submitting a syntactically invalid URL shows an error message and persists nothing.
4. Submitting a syntactically valid URL whose host is unreachable persists an Item with `fetch_status = "failed"` and shows a warning message.
5. The Library page lists every previously stored Item, newest first, after a page reload.
6. Killing and restarting the backend preserves every previously stored Item.

---

## 13. Out of Scope (Explicit Non-Goals)

The following **MUST NOT** be implemented in this iteration:

- User accounts, authentication, or authorization.
- Editing, deleting, archiving, or tagging Items.
- Full-text extraction of article bodies.
- Browser extensions, bookmarklets, or mobile clients.
- Background re-fetching or metadata refresh.
- Multi-user concurrency guarantees beyond TinyDB's defaults.
- Deployment, packaging, or production hosting concerns.

Features deferred here **MAY** be revisited in a follow-up spec.
